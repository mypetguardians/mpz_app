## 3. API 구조

### 3.1 API 엔드포인트 총괄

모든 API는 Django Ninja 기반이며 `/v1/` prefix 하에 동작합니다.

| 경로 | 앱 | 설명 |
|------|-----|------|
| `GET /` | - | Health Check (`{"ping": "pong"}`) |
| `/v1/auth/*` | user | 회원가입, 로그인, 토큰 갱신, 프로필 |
| `/v1/admin/*` | user | 관리자 전용 API |
| `/v1/kakao/*` | user | 카카오 OAuth 로그인/콜백 |
| `/v1/animals/*` | animals | 동물 CRUD, 목록, 필터, 이미지 |
| `/v1/adoptions/*` | adoptions | 입양 프로세스 전체 |
| `/v1/centers/*` | centers | 센터 관리, 동의서, 계약서 |
| `/v1/preset-questions/*` | centers | 프리셋 질문 템플릿 |
| `/v1/storage/*` | storage_service | 이미지 업로드/삭제 (Supabase Storage) |
| `/v1/favorites/*` | favorites | 즐겨찾기, 성격검사 |
| `/v1/notifications/*` | notifications | 알림 CRUD, 푸시토큰 |
| `/v1/posts/*` | posts | 게시글 CRUD, 태그, 좋아요 |
| `/v1/comments/*` | comments | 댓글/답글 CRUD |
| `/v1/ai/*` | ai | AI 추천 |
| `/v1/feedback/*` | feedback | 사용자 피드백 |
| `/v1/banners/*` | banners | 배너 관리 |
| `/v1/notices/*` | notices | 공지사항 |

### 3.2 인증 방식
```python
# api/security.py - JWTAuth
Authorization: Bearer <access_token>  # 헤더 방식
# 또는 쿠키의 accessToken/access 값
```
- 알고리즘: HS256
- Secret: `DJANGO_SECRET_KEY`
- access_token 만료: 1년 (365일)
- refresh_token 만료: 2년 (730일)

### 3.3 페이지네이션
```python
# api/pagination.py - CustomPageNumberPagination
Input:  page (1-based), page_size (기본 10)
Output: {
    "data": [...],
    "count": 100,        # 전체 개수
    "totalCnt": 100,
    "pageCnt": 10,       # 전체 페이지 수
    "curPage": 1,        # 현재 페이지
    "nextPage": 2,       # 다음 페이지 (없으면 null)
    "previousPage": null # 이전 페이지
}
```

### 3.4 API 문서
- URL: `/<engine>/` (staff 로그인 필요)
- 접근 제한: `@staff_member_required` 데코레이터

## 7. 동기화 스케줄러 (Worker 컨테이너)

Worker 컨테이너(`worker_entrypoint.sh`)에서 `run_scheduler.py` management command로 실행.
배포 시 worker는 재시작하지 않음 (동기화 중단 방지, `--no-recreate`).

### 7.1 스케줄

| 전략 | 스케줄 | 조회 범위 | 동작 |
|------|--------|----------|------|
| incremental | 매일 03:00 KST | 최근 2일 접수 | 신규 생성 + 업데이트 |
| status_sync | 매주 수/일 03:10 KST | 보호중 전체 | 기존 동물 상태 업데이트 + 보호종료 감지 (신규 생성 안 함, `update_only=True`) |
| full | 매월 1일 05:00 KST | 최근 90일 | 전체 재동기화 + 보호종료 감지 |

### 7.2 주요 특성
- **SyncLog 모델**로 이력 추적 — 시작/종료 시간, 전략, 결과(success/partial/fail), 통계
- **중복 실행 방지** — SyncLog에서 마지막 성공 날짜 조회, 같은 날 재실행 차단 (컨테이너 재시작 안전)
- **직접등록 동물**(is_public_data=False)은 동기화 영향 없음
- **보호종료 감지** — API에서 안 내려온 기존 "보호중" 동물을 자동으로 보호종료 전환

## 14. 주의사항

1. **PersonalityTest**: favorites 앱에 포함되어 있음 (별도 앱이 아님)
2. **공공데이터 동기화**: Worker 컨테이너(run_scheduler.py)에서 자동 스케줄 실행. 배포 시 worker는 `--no-recreate`로 재시작하지 않음 (동기화 중단 방지)
3. **WebSocket**: Redis가 없으면 WebSocket 기능 동작 안 함 (Docker Compose에 redis 서비스 포함)
4. **마이그레이션**: 배포 시 `migrate --noinput` 자동 실행됨 (GitHub Actions CI/CD에 포함)
5. **이미지 저장소**: Cloudflare R2에서 Supabase Storage로 이관 완료. 환경변수는 `STORAGE_*` prefix 사용
