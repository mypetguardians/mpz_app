## 2. AWS EC2 배포 환경

### 2.1 서비스 구성 (Docker Compose)

| 서비스 | 설명 | 이미지/Dockerfile | 포트 |
|--------|------|-------------------|------|
| Frontend (Next.js) | 웹 앱 | `frontend/Dockerfile` | 내부 (Nginx 경유) |
| Backend (Django) | API 서버 | `backend/Dockerfile` | 내부 (Nginx 경유) |
| Worker | 동기화 스케줄러 (run_scheduler) | `backend/Dockerfile` (동일 이미지) | - |
| Redis | WebSocket Channel Layer | `redis:7-alpine` | 6379 (내부) |
| Nginx | SSL termination, 리버스 프록시 | `nginx:latest` | 80, 443 |

Docker Compose 파일: `docker-compose.prod.yml` (dev/prod 공용, `APP_ENV`로 분기)

### 2.2 자동 배포 (GitHub Actions — GHCR 기반)

```
fork(WaterMinCho) → PR → upstream(mypetguardians) 머지 → upstream Actions 배포
```
- fork의 Actions는 비활성화. 배포는 upstream(mypetguardians/mpz_app) 리포지토리에서만 실행
- `git push origin main` → Prod EC2 자동 배포
- `git push origin dev` → Dev EC2 자동 배포

CI/CD 파이프라인 (`.github/workflows/deploy.yml`):
1. **빌드 단계**: GitHub Actions에서 FE/BE Docker 이미지 빌드 → GHCR(ghcr.io)에 push (EC2에서 빌드 안 함)
2. **배포 단계**: BE 환경변수 파일을 SCP로 EC2에 전송 (base64 디코딩) → EC2에서 이미지 pull → `docker compose up`
3. 마이그레이션 및 정적 파일 수집 자동 실행
4. Worker 컨테이너는 `--no-recreate`로 배포 시 재시작하지 않음 (동기화 중단 방지)

GitHub Secrets:
- `PROD_EC2_HOST`, `DEV_EC2_HOST`, `EC2_SSH_KEY` — EC2 접속
- `BE_ENV_PROD_B64`, `BE_ENV_DEV_B64` — BE 환경변수 (base64 인코딩)
- `KAKAO_CLIENT_ID` — 카카오 클라이언트 ID
- `GITHUB_TOKEN` — GHCR 로그인 (자동 제공)

### 2.3 수동 배포 (EC2에서)

```bash
cd ~/mpz_app

# 실행 (빌드 없이)
APP_ENV=prod docker compose -f docker-compose.prod.yml up -d

# 재빌드 (NEXT_PUBLIC_* 환경변수 반드시 커맨드라인으로 전달)
NEXT_PUBLIC_API_BASE_URL=https://api.mpz.kr/v1/ NEXT_PUBLIC_KAKAO_CLIENT_ID=e87b92ff4188fc038238a9a22eb0bf35 APP_ENV=prod docker compose -f docker-compose.prod.yml up -d --build
NEXT_PUBLIC_API_BASE_URL=https://dev-api.mpz.kr/v1/ NEXT_PUBLIC_KAKAO_CLIENT_ID=e87b92ff4188fc038238a9a22eb0bf35 APP_ENV=dev docker compose -f docker-compose.prod.yml up -d --build

# 상태 확인
APP_ENV=prod docker compose -f docker-compose.prod.yml ps

# 로그 확인
APP_ENV=prod docker compose -f docker-compose.prod.yml logs backend --tail=50
APP_ENV=prod docker compose -f docker-compose.prod.yml logs nginx --tail=20

# 마이그레이션
APP_ENV=prod docker compose -f docker-compose.prod.yml exec backend python manage.py migrate

# Nginx 재시작 (컨테이너 재생성 후 502 에러 시)
APP_ENV=prod docker compose -f docker-compose.prod.yml restart nginx
```

### 2.4 Nginx SSL

- SSL 인증서: Let's Encrypt (Certbot)
- 설정 파일: `nginx/dev.conf` (dev), `nginx/prod.conf` (prod)
- `APP_ENV` 환경변수에 따라 Docker Compose가 자동으로 올바른 conf 파일 마운트

### 2.5 주의사항
- `.env` 파일은 gitignore 대상. CI/CD에서 GitHub Secrets의 base64 인코딩 값을 SCP로 EC2에 전송
- Docker 빌드 시 `NEXT_PUBLIC_*` 환경변수는 `.env` 파일 방식은 안 먹힘. GHCR 빌드 시 build-args로 전달
- `docker compose restart`로는 env_file 변경 미반영. 반드시 `docker compose down && up -d` 또는 `--force-recreate` 사용
- EC2에서는 `docker compose` (v2) 사용 (`docker-compose` 아님)
- Worker 컨테이너는 배포 시 재시작하지 않음 (`--no-recreate`). 동기화 중단 방지 목적

## 15. 주의사항 및 보안

1. **Firebase 키**: 프로덕션에서는 반드시 환경변수(`FIREBASE_ADMIN_CREDENTIALS_JSON`)로 관리
2. **공공데이터 동기화**: Supabase Cron (pg_cron + pg_net)으로 자동 스케줄. 엔드포인트: `POST /v1/animals/sync-public-data`, 인증: `X-API-Key: {SYNC_API_KEY}` 헤더
3. **.env 파일 관리**: gitignore 대상. EC2에 직접 업로드해야 하며, CI/CD는 코드만 전송
4. **git push는 SSH로**: `git remote set-url origin git@github.com:WaterMinCho/mpz_app.git` (HTTPS 401 에러)
5. **Docker 빌드 시 NEXT_PUBLIC_* 환경변수**: `.env` 파일 방식은 안 먹힘. 반드시 커맨드라인으로 직접 전달
6. **Nginx 재시작**: 컨테이너 재생성 후 nginx가 IP 캐시 문제로 502를 줄 수 있음. `docker compose restart nginx` 필요
